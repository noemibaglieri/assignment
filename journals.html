User Experience Design

Today we talked about Conceptual Design. It is essential for a UX Designer to ask themselves what they want to create, what the assumptions of the idea are, and if it will achieve the result that is hoped for. To start, it’s important to put facts together by identifying usability and user experience goals and writing down initial user requirements based on findings (research, surveys, interviews…). Then proceed to write down claims and assumptions trying to defend and support them, this helps highlighting ones that are vague. After this we can start thinking about Conceptual Models, more specifically, outlining what users can achieve with a system and which concepts are essential to understand while interacting with it.

User Experience Design

This week we learned about “Use Cases”. We learned that these cases are important for expressing user interactions with the system we want to design. They can be presented by diagrams or simply by text which is easier to understand for the users. The diagrams help more when we need to represent a graphical view of the requirements, it also helps to identify system scope and boundary. While text-based “use cases” help describing the case in more depth. Either way, each use case needs a name, a number and a brief description.

Information Systems & Databases

In today’s lesson, we learned how to retrieve data from more than one table, how to use subqueries to combine select statements, set operators and joint notations. We now know that “joins” are used to link multiple tables to retrieve stored data, a “cartesian join” retrieves every combination of rows from the joint tables. There are two types of joins; “Inner Join” in which the records returned have at least one matching record in all tables and “Outer Join”, in which records can be returned even with no matching record. There are also “Self joins” which are used when a table needs to be joined with itself to retrieve the data required.

User Experience Design

Today’s lesson was about analysing data we gathered through questionnaires or interviews. We learned that is important to check for inconsistencies that might be wrong (data cleansing), as incorrect data will lead to incorrect results.
We learned about the three approaches; qualitative, quantitative and mixed-method. Quantitative data is the information we gathered that can be translated into numbers, this is why is easier to analyse. While qualitative is about statements given in open questions, notes from researchers, feelings. It is mostly interpretating results.
While one might be easier than the other to analyse, we learned that both methods are required for a system to be successful.

Information Systems & Databases

This week’s lesson we were taught how to create and delete tables, how to add constraints to a table, how to modify tables, how to insert, update and delete data from a table and we also learned about the purpose of views and their creations. Most of the time, for either security or other matters, it’s undesirable to have users access the entire table, so here’s when the “VIEW” mechanism comes in handy. User Experience Design

Today’s lesson was about Low-fidelity Prototype. Prototyping is very important for evaluating the design of an interactive product and to explore its suitability effectively. In UX Design a low-fidelity prototype can be a series of sketches, PowerPoint slide shows or cardboard mock-ups. Ideally, we want to approach a low-fidelity prototype to start with as it is quick, cheap and easily changed.

Information Systems & Databases

This week lecture was about database administration. We learned why concurrency control is essential, the nature and purpose of the transaction log, locking mechanisms and other concurrency protocols. We now know that a transaction is a sequence of database operations that need to be executed in their entirety or not at all. We also know it has ACID properties and the transaction log maintains a list of all changes so that database recovery is possible. We also learned that there’s various types of concurrency control.

User Experience Design

This week, we talked about High-Fidelity Prototyping. Unlike the low-fi one, this prototype looks more like the final product and has way more functionality and it is software-based. It is also useful for selling ideas to people. One of the downsides of this, is that when the user is testing a high-fi prototype they might think they’re working with a complete system, and this could be damaging for its reputation.
This type of prototype involves compromising, which we can divide in two types; horizontal (provides a wide range of functions with little details) and vertical which provides many details with only a few functions.
